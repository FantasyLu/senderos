# 路线绘图师（Route Mapper）

> 将游记、传记、人物行程转化为可交互的地理路线图。
> 核心输出：一个在浏览器中打开的 HTML 文件，内嵌 OpenStreetMap 底图、分层图钉、实线/虚线路径、时间轴侧边栏。

---

## 工具与资源速查

| 资源 | 路径 | 用途 |
|------|------|------|
| 地图 HTML 模板 | `references/map-template.html` | Leaflet 地图骨架，由 render_map.py 填充 |
| 古地名坐标库 | `references/ancient-place-names.md` | 历史地名 → 现代坐标映射，覆盖明清/唐宋/元代/玄奘/郑和/近现代欧非/南美等 |
| 电子书解析脚本 | `scripts/parse_ebook.py` | epub/mobi/azw3 → 结构化章节列表（纯文本+标题）|
| 坐标解析脚本 | `scripts/geocode.py` | 地名 → 坐标（优先本地库，次用 OSM Nominatim API）|
| 地图渲染脚本 | `scripts/render_map.py` | waypoints JSON → Leaflet HTML，含 CHECKPOINT 表格生成 |
| 地图输出目录 | `output/` | 所有生成的 HTML 文件存放于此 |

## Phase 0：入口分流

收到用户输入后，先判断路径：

| 用户输入 | 路径 |
|---------|------|
| 上传了 PDF，说「画路线」「根据这本书画地图」「画书中XX的路线」 | → **Mode A**（PDF 解析） |
| 上传了 epub / mobi / azw3，说「画路线」 | → **Mode A**（epub/mobi 解析） |
| 说「XXX走过哪些地方」「画出XX的路线」，无文件 | → **Mode B 纯检索** |
| 说「XX在YYYY-YYYY年间的路线」「XX某次旅行的路线」 | → **Mode B 纯检索**，同时记录时间段约束 |
| 说人物名 + 上传了 PDF/epub/mobi | → **Mode B 混合**（文件优先 + 网络补充）|
| 输入模糊（只说「画地图」）| → 追问：「是要根据一本书画路线，还是指定一个人物？」|

## Mode A：PDF / epub / mobi 解析模式

**支持格式**：`.pdf`、`.epub`、`.mobi`、`.azw3`、`.azw`

epub/mobi 文件先用 `parse_ebook.py` 转换为结构化章节列表，再按 PDF 解析流程处理：

```bash
python scripts/parse_ebook.py book.epub --output output/chapters.json
# 或
python scripts/parse_ebook.py book.mobi --output output/chapters.json
```

输出为章节 JSON（含 `chapter_num`、`chapter_title`、`text`、`word_count`），后续步骤与 PDF 完全一致。

### Phase A-0：确认目标人物

读取 PDF 后，**第一步先确认要画谁的路线**：

```
如果用户已明确指定人物（如「画诸葛亮的路线」）：
  → 记录目标人物，直接进入大部头预判

如果用户未指定（如「画这本书的路线」「根据这个 PDF 画地图」）：
  → 扫描 PDF 前几章，识别主要人物列表，询问用户：
    「这本书涉及以下人物，请问要画谁的行程路线？
      ① 作者/叙述者本人（如游记、自传）
      ② [人物A]
      ③ [人物B]
      …
      （也可以直接告诉我人物名）」

  用户选择后记录目标人物，继续。
```

**目标人物确认后，后续所有步骤只提取该人物的行程**，其他人物出现的地点不纳入。

**别名处理**：记录目标人物在书中的所有称呼（字、号、官职、代称等），提取时视为同一人。
例：诸葛亮 = 孔明 = 卧龙 = 丞相 = 武侯

---

### Phase A-0.5：大部头书籍预判

读取 PDF 后，估算体量：

```
如果章节数 > 20 或预估字数 > 10万：
  告知用户：
  「本书共约 N 章，篇幅较大。
   请问要处理全书，还是指定某几章？
   （可以先处理一个地区/时间段试试效果）」

  用户选择后继续；未回应则默认全书处理。
```

如处理范围已确认 → 将中间结果保存至 `output/<书名>_<人物名>_waypoints_draft.json`，支持续跑。

---

### Phase A-1：按章节分段

1. 识别章节边界（目录 / 标题 / 分隔符）
2. 如无明显章节结构：按约 4000 字切段，段间保留 500 字重叠（保证语义连贯）
3. 输出：`[章节1文本, 章节2文本, …]`，每段附章节编号和标题

---

### Phase A-2：逐章节提取地理信息

对每个章节，**只提取目标人物的行程**，其他人物经过的地点忽略。提取结构化地理信息，**输出必须严格符合以下 JSON 格式**，不得自由发挥：

**一章可以产出多个 waypoint**：若目标人物在同一章内移动至不同城市/地点，每个地点均单独建立节点，不要强行合并为一个，也不要只取最主要的一个。

```json
{
  "chapter_num": 3,
  "chapter_title": "第三章·滇西之行",
  "waypoints": [
      {
        "place_raw": "昆明府",
        "place_modern": "昆明",
        "layer": "major",
        "confidence": "confirmed",
        "route_to_next": "straight",
      "visits": [
        {
          "chapter_num": 3,
          "chapter_title": "第三章·滇西之行",
          "time_raw": "万历四十一年九月初一",
          "time_note": "约1613年10月",
          "duration": "三日",
          "description": "出发，休整补给三日",
          "source": "pdf",
          "source_sentence": "万历四十一年九月初一，余自昆明府启程……"
        }
      ]
    }
  ]
}
```

**字段规则（严格执行）**：

| 字段 | 规则 |
|------|------|
| `place_raw` | 原文中的地名，**原样保留**，不改写 |
| `place_modern` | **必填**。用你的知识将古地名翻译为现代地名或通用英文名，供坐标解析使用。格式：现代中文地名（如「昆明」「撒马尔罕」），对于中国境外或生僻地名，可填英文（如「Samarkand」「Tokmok, Kyrgyzstan」）。若完全不确定，填与 `place_raw` 相同的字符串。**此字段是坐标解析成功率的关键。** |
| `layer` | `"major"` = 有明确停留/住宿/重要事件；`"transit"` = 仅路过提及 |
| `confidence` | `"confirmed"` / `"single_source"` / `"inferred"` / `"disputed"`（见 Phase A-2.5）|
| `route_to_next` | `"described"` = 原文有路线描述；`"straight"` = 无描述，地图画虚线直线 |
| `time_raw` | 到达时间，**原样保留原文**，不改写。如原文无明确到达时间则留空 |
| `time_note` | 仅在换算有把握（有明确年号/公历锚点）时填写公历日期，否则**留空字符串** |
| `duration` | 停留时长，原文有描述时填写（如「三日」「约两周」），无描述留空字符串 |
| `source` | PDF 解析固定填 `"pdf"` |
| `source_sentence` | **必填**，原文依据句子（≤100字），防止 AI 杜撰地点 |
| `description` | 简述目标人物在此地的行为，≤50字，主语省略（默认为目标人物）|

**layer 判断标准**：
- `major`：住宿过夜 / 明确停留数日 / 发生重要事件（受病、遇盗、访古迹等）/ 作为出发或终点
- `transit`：「经过」「路过」「望见」「途经」，无停留描述

**多地点提取规则**：
- 若章节内出现移动动词（「来到」「抵达」「离开X前往Y」「翌日到达」等），且目的地为不同城市/地点，则分别建立节点
- 同一城市内的不同景点/街区**不拆分**（如在布哈拉游览三处清真寺，仍合并为一个布哈拉节点）
- 火车/飞机途中无停留的中间站，标记 `layer: transit`，仍应建立节点
- **判断依据必须是作者本人的一人称行为**（「我来到」「我们抵达」），背景叙事/历史介绍中提到的地名不建节点

### Phase A-2.5：准确性校验

提取完所有章节后，执行以下校验（**在坐标解析之前**）：

#### 步骤 1：source_sentence 反查
```
对每个 waypoint 的每个 visit：
  在对应章节原文中精确匹配 source_sentence 字段
  
  匹配成功 → 继续
  匹配失败（原文中找不到该句子）→
    标记该 visit: verification_status = "sentence_not_found"
    在 CHECKPOINT 中用 🔴 红色标注，提示人工确认
```

#### 步骤 2：置信度评级

为每个 waypoint 打上 `confidence` 字段（四级）：

| 级别 | 含义 | 条件 |
|------|------|------|
| `confirmed` | 多源确认或原文直接记载 | ≥2 个独立来源 OR source_sentence 匹配成功 + layer=major |
| `single_source` | 单一来源 | 仅 1 个来源，source_sentence 匹配成功 |
| `inferred` | 推断 | source 全为 "inferred" |
| `disputed` | 来源存在矛盾 | 不同来源对同一地点记载不一致 |

#### 步骤 3：时间逻辑校验
```
检查相邻主要节点时间顺序：
  若可提取时间数值 → 检查是否单调递增
  若发现倒序 → 在 description 注明「时间顺序疑似有误，请核实」

（无法解析时间的节点跳过，不强行判断）
```

---

### Phase A-3：跨章合并与去重

```
所有章节的 waypoints 按出现顺序合并
  ↓
同一地点合并：place_raw 完全相同 OR place_modern 相同
  → 保留第一次出现的坐标信息
  → 将每次到访的 visit 对象追加到 visits[] 数组中
  → visits 按章节顺序排列
  ⚠️  只合并跨章节的同一城市重复访问；同一章内的不同城市节点不合并
  ↓
坐标解析（调用 geocode.py）：
  python scripts/geocode.py 逐个地名解析
  — 先查 references/ancient-place-names.md（精选历史地名库）
  — 未命中 → 用 place_modern 字段的现代地名查 OSM Nominatim（间隔 ≥1.1 秒）
  — 仍失败 → 用 place_raw 原始地名再查一次 Nominatim
  — 全部失败 → 标记 resolved=false，收入「待确认列表」
```

调用 `python scripts/render_map.py <draft.json> --checkpoint` 生成确认表格，向用户展示：

```

主要节点：XX 处  途经点：XX 处
坐标解析成功：XX 处  坐标待确认：X 处

主要节点一览（可告知 AI 删除/修改任意行）：
| # | 地点 | 章节 | 时间 | 简述 | 层级 | 坐标 | 置信度 |
|---|------|------|------|------|------|------|--------|
| 1 | 昆明府 | 第三章·滇西之行 | 万历41年九月初一 | 出发，休整三日 | 主要 | ✅ | ✅ 确认 |
| 2 | 嵩明州 | 第三章·滇西之行 | 九月初四 | 途经 | 途经 | ✅ | 🟡 单一来源 |
| 3 | 龙井坡 | 第三章·滇西之行 | 九月初五 | 住宿一晚 | 主要 | ⚠️ 待确认 | 🔴 原文句未找到 |
…

⚠️ 以下地名无法自动解析坐标：
  - 「龙井坡」「三台山」…

```

**等用户确认（或修改）后，才进入渲染。**

确认请求格式：

```

📋 地图基本信息确认：
  标题：<meta.title>
  副标题：<meta.subtitle>
  （依据：首节点 time_raw=「…」，末节点 time_raw=「…」，作者/人物=「…」）

如需修改标题或副标题，请一并告知，确认后进入渲染。
```

### Phase A-4.5：节点密度判断与显示方案选择

坐标确认后，统计有效节点数（resolved=true 的节点），按阈值给出建议：

```
节点数 ≤ 30：
  → 直接渲染，全部显示，无需用户选择

节点数 31–80：
  → 向用户展示选项：
    「共提取 N 个有效节点，地图可能较为密集。建议选择显示方式：
      ① 全部显示 + 聚合模式（相近图钉自动合并，缩放后展开）【推荐】
      ② 仅显示主要节点（major），途经点隐藏
      ③ 按章节分段显示（顶部切换器，每次只看一章）」
  → 等用户回复，默认选 ①

节点数 > 80：
  → 强烈建议分段：
    「共 N 个节点，建议按章节分段显示，否则地图会非常拥挤。
      ① 按章节分段显示（推荐）
      ② 强行全部显示 + 聚合模式」
  → 等用户回复，默认选 ①
```

用户选择结果传入渲染命令：
- 选 `①全部+聚合`：`--cluster`
- 选 `②仅主要节点`：`--major-only`
- 选 `③/①分段显示`：`--stages`（按章节分组，顶部切换器）

### Phase A-5：生成地图

渲染前，先将 `meta` 字段写入 waypoints JSON（title/subtitle 存在数据中，无需命令行参数）：

```json
{
  "meta": {
    "title": "<见下方规则>",
    "subtitle": "<见下方规则>"
  },
  "waypoints": [...]
}
```

**`meta` 字段填写规则（填写前必须抽样读取 ≥3 个节点的 `chapter_title` 和 `time_raw` 判断性质）**：

| 场景 | title | subtitle |
|------|-------|----------|
| 游记/传记（作者即主角）| `《书名》· 作者路线` | `作者名 著 · 起止年`（无年份则只写 `作者名 著`）|
| 历史人物（无明确作品）| `人物名行程路线` | `朝代/时代 · 起止年` |
| 文学/影视作品中的虚构人物 | `《作品名》· 人物名路线` | `作者名 著` 或 `出版年/年代` |

- **时间跨度**：从 waypoints 首尾有效 `time_raw` 中提取；`time_raw` 普遍为空时省略
- **时间跨度含义**：行程发生的时间，而非作品出版时间（历史人物检索模式除外）
- ⚠️ 禁止凭文件名或书名联想填写；必须有节点数据或用户陈述为依据

```bash
python scripts/render_map.py output/<书名>_waypoints_final.json \
  output/<书名>_路线图.html \
  [--cluster]      # 启用 markercluster 聚合
  [--major-only]   # 仅渲染主要节点
  [--stages]       # 启用阶段切换器（按章节）
```

告知用户：「地图已生成：`output/<书名>_路线图.html`，用浏览器打开即可查看。」

## Mode B：人物检索模式

### Phase B-0：输入确认

收到人物名后，确认：

1. **有无 PDF 辅助资料**：「你有没有关于TA的书籍、文献、游记 PDF？有的话直接上传，比网络搜索质量高」
2. **人物类型**：历史人物 / 现代人物（影响信息源选择策略）
3. **路线范围**（可选）：全生涯 / 某次具体旅程 / 某个时间段

**时间段参数提取**：若用户输入中包含时间约束（如「1920-1930年间」「二战期间」「青年时期」），记录为：

```
TIME_RANGE = {
  "start": 1920,   # 开始年份（整数，若为模糊描述则估算）
  "end": 1930,     # 结束年份（整数）
  "raw": "1920-1930年间",  # 用户原始描述，原样保留
  "is_estimate": false     # 若年份由模糊描述估算而来，标为 true
}
```

若无时间约束，`TIME_RANGE = null`，默认处理全生涯路线。

**确认不阻塞交付**：如用户直接说「画 XXX 的路线」，默认全生涯路线 + 无 PDF + 自动判断历史/现代，直接进入 Phase B-1。

---

### Phase B-1：信息采集（三路并行 Agent）

**如有用户提供的 PDF** → 先标记为 `source: "user-pdf"`，优先解析；网络检索只针对 PDF 信息缺口补充。

**信息源优先级（高 → 低）**：
1. 用户提供的 PDF（`source: "user-pdf"`）
2. 书籍/传记/历史文献（`source: "pdf"`）
3. 网络检索（`source: "web"`）
4. AI 基于历史背景的合理推断（`source: "inferred"`）

并行启动三个 Agent：

#### Agent 1：PDF 资料分析（如有）
```
如果用户提供了 PDF：
  按 Mode A 的章节分段 + Phase A-2 的提取格式处理
  标注 source: "user-pdf"
  识别 PDF 覆盖了哪些时间段/路段
  输出：已提取的 waypoints + 信息缺口列表
如果无 PDF：
  此 Agent 跳过，直接由 Agent 2 全面搜索
```

#### Agent 2：网络检索（补充缺口）
```
搜索目标（针对 Agent 1 的信息缺口；若无 PDF 则全面搜索）：
  - 人物传记、回忆录、日记、游记原文
  - 重要旅行/出行记录（访谈、年谱、大事记）
  - 历史文献中关于此人行踪的记载

如果 TIME_RANGE != null：
  - 搜索关键词中加入时间约束（如「玄奘 629-645 行程」「鲁迅 1920-1930 活动地点」）
  - 优先检索该时间段的年谱、大事记、传记对应章节
  - 对检索到的每条信息，尝试判断其时间是否落在 TIME_RANGE 内；
    无法判断时间的条目标记 time_uncertain=true，保留供 Agent 3 处理

信息源偏好：
  历史人物 → 正史、方志、文集、考古研究
  现代人物 → 自传、采访、旅行类著作

信息源排除：
  - 知乎、微信公众号、百度百科（可信度低）
  - 纯转述/二手总结（优先找原文）

输出格式：与 Phase A-2 相同的 waypoints JSON，source 字段填 "web"
```

#### Agent 3：时间线梳理与交叉验证
```
整理行程时间轴（按年份/阶段）
交叉验证：PDF 内容 vs 网络资料是否有矛盾
古地名解析：查询 references/ancient-place-names.md（历史人物必做）
发现矛盾时：保留矛盾，在 visit.description 中注明「来源存在分歧」

如果 TIME_RANGE != null，对每个 waypoint 执行时间过滤：
  对每个 visit，尝试从 time_raw 解析年份：
    - 能解析出年份 → 判断是否在 [TIME_RANGE.start, TIME_RANGE.end] 内
      · 在范围内 → in_range: true
      · 不在范围内 → in_range: false
    - 无法解析年份（time_uncertain=true）→ in_range: "uncertain"，保留但在 CHECKPOINT 中标注
  
  过滤规则：
    · 某 waypoint 的所有 visit 均为 in_range: false → 整个节点标记 excluded: true，不纳入最终路线
    · 某 waypoint 有部分 visit in_range: false → 仅保留 in_range: true/uncertain 的 visit
    · 被排除的节点汇总为「时间范围外节点列表」，在 CHECKPOINT 中展示

输出：按时间排序的完整 waypoints 列表，每个 visit 含 in_range 字段
```

```
合并三个 Agent 的结果
  ↓
按时间顺序排列
  ↓
推断填补（仅在信息量足够支撑合理推断时执行）：
  如果 A 地到 B 地之间有信息空白，且有史料/逻辑支撑路径推断

  ↓

```

## 地图规范（渲染标准）

### 图钉颜色与样式

| 类型 | 颜色 | 说明 |
|------|------|------|
| 主要节点（confirmed） | 深橙色实心 | 多源确认或原文直接记载 |
| 主要节点（single_source） | 橙色实心 + 橙框 | 单一来源 |
| 主要节点（inferred） | 金色实心 | 全为推测来源 |
| 主要节点（disputed） | 红橙色实心 + 问号角标 | 来源存在矛盾 |
| 途经点（有记载） | 蓝灰色小圆点 | 路过但无停留 |
| 途经点（推测） | 橄榄色半透明 | 推测途经 |

### 路线样式

所有相邻节点之间以流动动画路径连接，无实线/虚线区分。路线只连接有坐标的相邻节点（按 `order` 顺序）。

### 图钉 Popup（点击弹出）

主要节点 Popup 内容（多次到访分块展示）：

```
📍 昆明府
   昆明（现代地名）
━━━━━━━━━━━━━
第1次到访
  📖 第三章·滇西之行
  🕐 到达：万历四十一年九月初一（约1613年10月）
  ⏱ 停留：三日
  出发，休整补给
第2次到访
  📖 第十一章·返程记
  🕐 到达：十一月下旬
  返程途经
```

### 时间轴侧边栏

- 仅显示 major 节点
- 按章节/时间顺序排列
- 每个节点显示：地名、到达时间（若有）、停留时长（若有）
- 点击侧边栏条目 → 地图飞到对应地点并弹出 Popup
- 多次到访标注「（N次到访）」

## 失败处理（if-then 三段式）

| 触发条件 | 一线修复 | 仍失败兜底 |
|---------|---------|-----------|
| PDF 无法提取文字（扫描图片版）| 提示用户「此 PDF 为图片格式，无法直接提取文字。请提供文字版，或手动列出地名列表」| 用户提供地名列表后走手动模式 |
| 章节无明显地理信息 | 跳过该章节，在报告中注明「第X章未发现地理信息」| 继续处理其他章节 |
| 地名无法解析坐标 | CHECKPOINT 中列出，询问用户处理方式 | 默认「保留但标注位置待确认」，在地图上用特殊样式显示 |
| PDF 与网络资料地点有矛盾 | 保留矛盾，description 注明「来源存在分歧：PDF记载为X，网络资料称Y」| 以 PDF 为准（用户提供的资料权重更高）|
| 网络检索信息极少（<5条可用来源）| 告知用户信息不足，降低期望；以 PDF 为主，空白路段用推测填补并显著标注 | 生成仅含 PDF 信息的地图，推测路段全部虚线 |
| Agent 超时/搜索无结果 | 继续推进，Phase B-2 标注「信息不足」，CHECKPOINT 中报告 | 该路段用虚线直线连接，不添加推断节点 |
| 大部头书（>20章）处理中断 | 已提取章节写入 `output/<书名>_<人物名>_waypoints_draft.json`，下次续跑时先读取该文件 | 用户可以选择只处理已完成的部分先生成地图 |
| PDF 中目标人物行程与其他人物行程交织难以区分 | 以 `source_sentence` 为准，句中主语必须是目标人物或其别名才纳入 | CHECKPOINT 中标注「主语不确定」，让用户逐条确认 |
| PDF 中人物众多，无法自动识别主要人物列表 | 提示用户直接说出目标人物名 | 进入手动模式，用户逐章指定 |
| `render_map.py` 执行失败 | 检查 Python 版本（需要 3.9+）；检查模板文件是否存在 | 输出原始 waypoints JSON，告知用户手动替换模板中的占位符 |

## 反模式黑名单（绝不做的事）

| # | 禁止行为 | 替代做法 |
|---|---------|---------|
| 1 | 无法解析坐标时静默跳过，不告知用户 | 必须在 CHECKPOINT 中列出所有 unresolved 地名 |
| 2 | 把推测路线画成实线（误导用户认为是记载）| 推测路段强制虚线，Popup 中标注 `[推测]` |
| 3 | 原文没有的地名，AI 自行「补充」添加 | 必须有 `source_sentence` 原文依据，查不到原文依据的地名不添加 |
| 4 | 时间换算没把握时强行换算，产生错误日期 | `time_note` 留空，保留原文 `time_raw` |
| 5 | PDF 和网络资料有矛盾时强行合并 | 保留矛盾，description 注明分歧，以用户 PDF 为准 |
| 6 | 一次性跑完所有内容才让用户看到结果 | 必须在坐标解析后、渲染前设置 CHECKPOINT |
| 7 | 大部头书不询问范围直接全量处理 | 超过 20 章时先询问用户要处理的范围 |
| 8 | 用知乎、微信公众号、百度百科作为人物信息来源 | 只用权威媒体、正史、传记、学术资料 |
| 9 | transit 途经点不参与路线连线（旧规则，已移除） | 现在所有节点按时间顺序依次连线，transit 节点也参与连线但使用虚线 |
| 10 | 同一地点多次出现时创建多个图钉重叠 | 合并为一个图钉，visits[] 数组保存所有到访记录 |
| 11 | PDF 中其他人物到过的地点混入目标人物路线 | source_sentence 的主语必须是目标人物或其别名，否则不纳入 |
| 12 | 用户未指定人物时，默认假设是作者路线 | 必须先询问确认目标人物，再开始提取 |
| 13 | 凭文件名、书名、人物名联想填写 `meta.title` / `meta.subtitle` 等描述性字段 | 填写前必须抽样读取 ≥3 个节点的 `chapter_title` 和 `time_raw`，判断年代和叙事性质后再填写 |
